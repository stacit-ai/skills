# Training Plus Service Product Projects

Use this branch when one repo must keep training/evaluation code and inference/service
code versioned together. Treat it as a product project with separated subprojects and a
shared harness.

## Structure

Create the repo root, then run `uv init` inside Python subprojects such as `training/`.
Copy the harness/tool files from
[assets/training-service-product-template](../assets/training-service-product-template)
after initialization. Adapt service structure only after the serving approach is known.

Default shape:

- root `AGENTS.md`
- root `.agents/knowledge/WORKFLOW.md`
- root `.agents/knowledge/QUALITY.md`
- root `.agents/knowledge/REFERENCES.md`
- root `.agents/knowledge/GOALS.md`
- root `justfile`
- root `.pre-commit-config.yaml`, when Git hooks are selected and composed for this repo
- `training/` as a reproducible training/evaluation subproject
- `service/` as a framework-neutral service boundary when service work starts
- `tests/integration/` for train/inference consistency checks when needed

## Pre-Commit Composition

Generate one root `.pre-commit-config.yaml` from the selected root and subproject tools;
do not copy a universal config. Order non-security hooks by nearest, cheapest failure,
then run secret scanning last:

1. Root basic file rules and dependency/lock consistency for each managed subproject.
2. Per-subproject formatting, then linting, limited by file filters or local commands.
3. Per-subproject type checking only where explicitly configured.
4. Fast unit and tiny train/service contract tests only when stable and consistently
   quick.
5. One root secret scanner, Gitleaks by default; copy its hook block from
   [references/secret-scanning.md](secret-scanning.md) and keep it last.

Never include GPU work, model downloads, artifact builds, training, full integration,
or end-to-end service tests. Root `justfile` recipes must expose format, lint, tests,
integration checks, and aggregate checks as direct commands; none may delegate the
whole check suite to pre-commit.

## Harness Rules

- Root harness owns shared goals, cross-project workflows, references, CI expectations,
  security rules, and train/inference consistency requirements.
- Use `.gitignore` files generated by `uv init` in Python subprojects; append
  root-level or cross-project ignored outputs only when those paths are created.
- `training/AGENTS.md` owns training-specific layout, data, config, and workflow rules.
- `service/AGENTS.md` owns serving-specific layout, runtime, API, robustness, and
  deployment rules after the service framework is chosen.
- Keep shared conventions in the root knowledge base so training and service evolve
  together.
- Do not duplicate long-lived instructions in both subprojects unless local override is
  required.

## Training Side

Apply the reproducible training/evaluation rules:

- `uv` project by default.
- Code under the training package created by or after `uv init`.
- The asset bundle contains no `pyproject.toml`. Merge pyproject tool configuration from
  [references/reproducible-training.md](reproducible-training.md) into the training
  subproject's file generated by `uv init`.
- Data layers under `training/data/`.
- Lightweight CI and manual-only slow training checks.
- Early failure for unexpected training/evaluation states.
- Apply [references/pytorch.md](pytorch.md) when training uses PyTorch, and record which
  backend extra each training, CI, container, and service environment installs.

## Service Side

Service code has stronger long-term maintenance and robustness needs than training
scripts. However, do not choose a framework by default. Create only a service boundary
and local harness until requirements determine whether the service should use FastAPI,
BentoML, TorchServe, a batch runner, a queue worker, or another interface.

After a framework is selected, write its rules into `service/AGENTS.md` and root
knowledge files. Future agents cannot infer service requirements from this skill.

## Integration

Add placeholders or guidance for integration checks that protect train/inference
consistency:

- model artifact format compatibility
- preprocessing and postprocessing parity
- tokenizer or feature schema compatibility
- representative inference on a tiny fixture
- versioned config or metadata handoff

Keep default CI lightweight. Run only cheap consistency tests automatically; mark
artifact-building, GPU, or end-to-end service tests as manual commands.
