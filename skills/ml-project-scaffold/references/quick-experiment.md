# Quick Experiment Projects

Use this branch for one-off or short-lived ML work whose goal is to test an idea or
show feasibility. Optimize for readable, editable scripts and low setup cost.

## Structure

Run `uv init` first, then copy the harness/tool files from
[assets/quick-experiment-template](../assets/quick-experiment-template). Create
directories and workflow files only when the experiment needs them.

Common root files:

- `AGENTS.md`
- `justfile`
- `requirements.in`
- `requirements.dev.txt`
- generated `requirements.txt`
- `.gitignore`
- `.pre-commit-config.yaml`, only when Git hooks are selected and composed for this
  project
- `train.py`, `eval.py`, and other workflow entry files when needed

Common directories:

- `configs/` for coarse experiment settings, when needed
- `data/` for raw immutable inputs, when needed
- `notebooks/` for human exploration, when needed
- `outputs/` for logs, checkpoints, predictions, and run artifacts, when needed
- `scripts/` only when a command becomes too long or complex for `justfile`
- `tests/` only when custom code needs protection
- `<project_name>/` only for shared logic that multiple workflows must use consistently

## Dependency And Command Defaults

- Initialize with `uv init`, then use `requirements.in` for the quick-experiment
  dependency flow.
- Put only implementation/runtime dependencies in `requirements.in`.
- Put `pre-commit` in `requirements.dev.txt` by default, alongside selected development
  tools such as `ruff` and `pytest`. Do not lock this file; install latest compatible
  tool versions during setup.
- Use the `.gitignore` generated by `uv init`; append project-specific ignored outputs
  such as `outputs/`, checkpoints, or local data products when those paths are created.
- Document a compile command equivalent to:
  `uv pip compile requirements.in --output-file requirements.txt --no-annotate --no-emit-index-url`
- Keep `requirements.txt` readable: version pins only, no hashes, no dependency origin
  comments. Install it with
  `uv pip install -r requirements.txt --torch-backend=auto`; this keeps the common
  local PyTorch case convenient without forcing `torch` into every project.
- `justfile` default target prints command help and includes commands for environment
  creation, dependency compile/install, direct formatting and linting, training, and
  evaluation. Add tests to the default check only after the project creates tests.

## Pre-Commit Composition

Generate a small `.pre-commit-config.yaml` for the tools this experiment uses; do not
copy a universal config. Order non-security hooks by nearest, cheapest failure, then
run secret scanning last:

1. Basic file rules: trailing whitespace, EOF, YAML syntax, merge conflicts, large
   files, and line endings.
2. Requirements consistency with the uv `pip-compile` hook because this branch commits
   `requirements.in` and `requirements.txt`.
3. Ruff formatting, then Ruff linting.
4. One secret scanner, Gitleaks by default; copy its hook block from
   [references/secret-scanning.md](secret-scanning.md) and keep it last.

Do not add type checking or tests by default. Add a fast test hook only after tests
exist and consistently complete quickly, and place it before the final secret scanner.
Never add training, GPU, model-download, or smoke-test commands. Keep `just format`,
`just lint`, and `just check` as direct tool commands rather than wrappers around
pre-commit.

## Harness Rules

Use one concise root `AGENTS.md`. If the rules no longer fit in a short file, stop and
reconsider whether the project should be a reproducible training project instead.

The quick-experiment harness should cover:

- file structure
- intended outcome
- useful reference links
- Conventional Commits without scope by default
- command usage
- data and output rules
- code style and documentation expectations
- repo safety checks such as secret scanning and env-file exclusion

Do not add a thick knowledge base, heavy CI, or strict architecture unless the user
explicitly asks.

If PyTorch is used through this requirements-based flow, read
[references/pytorch.md](pytorch.md) and document that automatic backend selection is a
local convenience rather than a reproducible deployment target.

## Code Rules

- Prefer procedural workflow entry files with a clear `main`.
- Workflow files are final entrypoints and should not be imported.
- Put shared code in the project module only when divergent copies would create logical
  inconsistency.
- Do not extract helpers merely to reduce repetition.
- Add docstrings to shared module code. Public or workflow-called interfaces document
  parameters, returns, and errors; private helpers can use a one-line purpose docstring.
- Use comments for long logical blocks, not for obvious individual statements.
- Fail early for unexpected errors. Handle dirty data defensively only when there is a
  known reason and the alternative would waste an expensive run.

## Data, Config, Outputs, Tests

- `data/` is raw immutable input. No workflow except a deliberate data-preparation step
  should write there.
- `outputs/` is ignored by git. Code must not require it to already exist.
- Default config style is `pydantic_settings` with TOML when configuration is useful.
- Config files should contain values expected to change between experiments, not a
  one-to-one modular assembly of the codebase.
- `tests/` is optional and targeted. Add tests only for custom components whose
  correctness matters.
