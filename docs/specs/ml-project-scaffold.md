# Spec: ml-project-scaffold

## Purpose

Guides an agent through initializing a machine learning project with an appropriate
file structure, command runner, dependency strategy, and harness layer. The skill
exists because ML projects vary sharply by lifecycle: a quick experiment should stay
thin and permissive, a reproducible training project needs durable conventions, and a
training-plus-service product needs separated subprojects with shared knowledge.

The skill must make future project agents self-sufficient. After initialization, agents
working inside the created project will not see this skill, so every lasting code,
architecture, data, quality, and command convention must be written into the created
project's harness files.

## Trigger Conditions

Explicit:

- "create an ML project"
- "initialize a machine learning repo"
- "scaffold a training project"
- "set up an ML experiment repo"
- "create a training and inference service project"
- "add ML project harness files"

Indirect:

- "I need a repo for a quick model experiment"
- "make this training work reproducible"
- "set up files for training and evaluation"
- "start a project that keeps training and serving consistent"
- "create the justfile, uv setup, and agent instructions for an ML project"

## Scope

In scope:

- Classifying the project as quick experiment, reproducible training/evaluation, or
  training-plus-service product.
- Initializing the Python project with `uv init`, copying the corresponding
  harness/tool templates, and creating only the directories the project needs.
- Establishing dependency and command-runner defaults.
- Capturing ML-specific data, output, configuration, testing, and quality rules in the
  target repo's agent-facing files.

Out of scope:

- Implementing model training, evaluation, data processing, or service business logic.
- Binding the service product harness to a specific serving framework by default.
- Creating documentation websites by default.
- Enforcing heavy type checking for ML framework code by default.

## Defaults

- Skill name: `ml-project-scaffold`.
- Command runner: `justfile`.
- Quick experiments: `uv` plus `requirements.in` for runtime dependencies and
  unpinned `requirements.dev.txt` for development tools; document how to compile a
  version-pinned `requirements.txt` without hashes, annotations, or dependency comments.
  Install it with `--torch-backend=auto` by default without adding an undeclared Torch
  dependency.
- Reproducible training projects: use the `pyproject.toml` generated by `uv init`;
  asset bundles intentionally omit this file, so merge tool configuration from the
  reference into the generated project metadata. Keep type annotations for IDE help but
  do not enable type checking by default.
- Training-plus-service projects: framework-neutral service harness; select a service
  framework only after project requirements justify it.
- Quick experiment harness: one root `AGENTS.md`, repo safety checks only.
- Reproducible and product harness: root `AGENTS.md` plus `.agents/knowledge/` files,
  `.editorconfig`, CI, lightweight tests, and manual-only expensive checks. Compose
  pre-commit from selected tools when Git hooks are needed; Gitleaks is the default
  scanner when secret blocking is required.
- Asset bundles omit platform-specific CI/CD files. Generate CI only after selecting
  GitHub Actions, GitLab CI/CD, or another target platform.
- Development dependencies include `pre-commit` by default for every lifecycle class,
  but asset templates do not include a universal `.pre-commit-config.yaml`. Each
  lifecycle reference defines its own hook thickness and ordering, with secret scanning
  always last after file-changing hooks and other checks.
- `justfile` and CI run quality tools directly; they do not use pre-commit as an
  aggregate command runner.
- PyTorch projects: determine the target OS, Python, accelerator, and driver/runtime
  matrix before adding dependencies. Reproducible uv projects use mutually exclusive
  backend extras by default; quick `uv pip` experiments may use local automatic backend
  selection.
- `.gitignore` files are generated by `uv init`; templates do not include `.gitignore`
  files and references tell agents to append project-specific ignores only when needed.

## Quality Bar

The skill succeeds when an agent can initialize the correct ML project class with
`uv init`, preserve the intended harness thickness, copy the appropriate harness/tool
templates, create the needed project structure, and leave the generated repo with
enough agent-facing instructions that future agents do not need this skill.
